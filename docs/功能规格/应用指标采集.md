# WeMonitor — 应用指标采集

> 状态：已实现

## 1. 功能概述

支持两种模式采集被监控服务的应用级指标：
- **Pull 模式**：WeMonitor 定时拉取服务的 Prometheus `/metrics` 端点
- **Push 模式**：外部服务主动 POST 指标到 WeMonitor

采集到的指标统一存入 SQLite `metrics` 表，与其他系统指标共用存储和查询体系。

## 2. Pull 模式

### 2.1 工作流程

```
WeMonitor 定时任务
  → HTTP GET <scrape_url>
  → 解析 Prometheus text format
  → 批量写入 SQLite metrics 表
```

### 2.2 配置方式

在「服务管理」页为服务设置 `scrape_url`（如 `http://127.0.0.1:5174/metrics`）和 `scrape_interval`。

### 2.3 Prometheus 解析

`lib/collector/scraper.js` 解析 Prometheus 标准格式：
- Counter / Gauge / Histogram 类型的指标
- 支持标签（labels）
- 忽略 `# HELP` / `# TYPE` 注释行

## 3. Push 模式

### 3.1 接口

**`POST /api/v1/metrics`**

**鉴权**：`X-API-Key` header（在 `config.js` 中配置）

**请求体**：
```json
{
  "service": "backup-script",
  "metrics": [
    { "name": "backup_duration_seconds", "value": 42.3, "labels": { "target": "music_db" } },
    { "name": "backup_size_bytes", "value": 1048576 },
    { "name": "backup_success", "value": 1 }
  ]
}
```

### 3.2 适用场景

- 短期任务（备份脚本、数据处理脚本）
- 无法暴露 HTTP 端点的服务
- 外部定时上报的场景

## 4. 指标查询

与系统指标共用查询 API（`/api/v1/metrics` 和 `/api/v1/metrics/batch`），通过 `service` 参数区分数据来源。

详见 [系统资源监控](系统资源监控.md) 第 3.3 节。

## 5. 关键文件

| 文件 | 用途 |
|------|------|
| `lib/collector/scraper.js` | Pull 模式采集 + Prometheus 解析 |
| `lib/api/metrics.js` | 指标查询 + Push 上报 API |
| `routes/api.js` | API 路由（含 X-API-Key 鉴权） |
