# WeMonitor — 事件时间线

> 状态：规划中

## 1. 背景

当前系统中分散着多种事件（部署、健康检查变化、数据清理、备份、cron 任务执行），但缺乏一个统一视图来纵览系统近期发生了什么。

## 2. 功能需求

### 2.1 统一时间线

一个页面聚合以下事件类型：

| 事件类型 | 数据来源 | 显示内容 |
|---------|---------|---------|
| 部署事件 | `deploy-events.jsonl` | 服务名、版本、阶段、结果 |
| 健康检查变化 | `health_checks` 表 | 服务名、状态变化（healthy ↔ unhealthy） |
| 服务配置变更 | 新增/编辑/删除/启停服务 | 操作人、操作类型、对象 |
| 数据清理 | 清理器日志 | 清理的数据量和时间范围 |
| 备份事件 | `backup_config` 表 | 服务名、备份时间、大小 |
| Cron 任务执行 | `cron-history.jsonl` | 任务名、exit code、耗时 |

### 2.2 时间线 UI

类似 GitHub / GitLab 的风格：

```
2026-07-21 14:30
  🔵 WeMusic 部署成功 (abc1234 → def5678, 45s)
2026-07-21 14:25
  🔴 WeMusic 健康检查失败 (TCP 超时)
2026-07-21 14:23
  🟢 WeMusic 恢复
2026-07-21 12:00
  🗑 数据清理: 删除 720 条 1min 数据 (2026-07-14 ~ 2026-07-21)
```

- 颜色编码：🟢 正常 / 🔵 信息 / 🟡 警告 / 🔴 异常
- 事件分组：同一类型 + 相近时间（≤ 2 分钟）合并为一条
- 默认展示最近 24 小时，可切换 7 天 / 30 天
- 按事件类型筛选（checkboxes）

### 2.3 无限滚动/分页

- 首次加载最近 50 条
- 滚动到底部加载更早的
- 或使用传统分页

## 3. 技术方案

### 3.1 事件聚合

`lib/api/timeline.js` — 从多个数据源拉取事件，按时间戳合并排序：

```js
async function getTimeline(since) {
  const deployEvents = readDeployEvents(since);
  const healthEvents = await db.query('SELECT ... FROM health_checks WHERE ...');
  const cronEvents = readCronHistory(since);
  // ...
  return [...deployEvents, ...healthEvents, ...cronEvents]
    .sort((a, b) => b.timestamp - a.timestamp);
}
```

### 3.2 数据量控制

- 各数据源独立限制条数（如每种最多 200 条）
- 总返回不超过 500 条
- 支持游标分页（`before=<timestamp>`）

## 4. API 端点

| 方法 | 端点 | 说明 |
|------|------|------|
| `GET` | `/api/v1/timeline` | 事件列表（支持 since/before/limit/types 参数） |
