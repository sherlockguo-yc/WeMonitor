# WeMonitor — Status Page（公开服务状态页）

> 状态：规划中

## 1. 背景

当前服务健康状态需要登录才能查看。提供一个无需认证的公开页面，可视化展示各服务运行状态，适合发给用户或嵌入其他页面。

## 2. 功能需求

### 2.1 公开访问

- 独立路由（如 `/status`），无需登录
- 可选的密码保护（环境变量配置，不设密码则完全公开）

### 2.2 展示内容

| 区域 | 内容 |
|------|------|
| 页头 | 标题、最后更新时间、整体状态摘要 |
| 服务状态卡片 | 每个服务一张卡片：名称、状态点（绿/红/灰）、响应时间 |
| 系统指标 | CPU/内存概览（可选，受密码保护） |
| 事件时间线 | 最近 24 小时的故障/恢复/部署事件 |
| 页脚 | 刷新间隔、Powered by WeMonitor |

### 2.3 状态定义

| 状态 | 图标 | 说明 |
|------|------|------|
| Operational | 🟢 | 所有检查通过 |
| Degraded | 🟡 | 部分检查失败或延迟偏高 |
| Outage | 🔴 | 不可达 |
| Unknown | ⚪ | 无数据 |

### 2.4 自动刷新

- 页面 30 秒自动刷新
- 也可手动点击刷新按钮

### 2.5 响应式

- 桌面：卡片网格（3-4 列）
- 移动端：单列堆叠

## 3. 技术方案

### 3.1 路由

`GET /status` → `views/status.ejs`，无需 auth 中间件。

与主站共享数据（同一 SQLite + 同一 API），只是渲染方式不同。

### 3.2 数据获取

两种方式：
- **A（推荐）**：服务端渲染时查询数据库，直接注入 EJS 模板
- **B**：纯静态页面 + JS fetch 已有 API（`/api/v1/health`、`/api/v1/stats/current`），但需要这些 API 对未登录用户开放

推荐方案 A——独立的公开路由，数据从后端计算后直接渲染，不需要暴露内部 API。

### 3.3 可选密码保护

```js
// config.js
statusPagePassword: process.env.STATUS_PAGE_PASSWORD || null
```

如果设置了密码：
- 访问 `/status` → 输入密码页面
- 密码验证通过后设置临时 cookie（1 小时），后续无需再输入
- 不设密码则直接展示

### 3.4 样式

独立于管理后台的布局，不需要侧栏。更简洁的展示型页面：
- 居中容器，最大宽度 960px
- 卡片式布局
- 品牌色 / Logo（可选）

## 4. API 端点

| 方法 | 端点 | 说明 | 鉴权 |
|------|------|------|------|
| `GET` | `/status` | Status Page | 公开（可选密码） |
| `GET` | `/api/v1/status/data` | Status 页面数据 | 公开（可选密码） |
