# WeMonitor — 进程查看

> 状态：规划中

## 1. 背景

当前 Dashboard 只有宏观系统资源视图，缺少进程级别的视角。管理员需要了解哪些进程在运行、各自的资源占用。

## 2. 功能需求

### 2.1 进程列表

表格展示所有进程，默认按 CPU 使用率降序：

| PID | 进程名 | CPU% | 内存(MB) | 运行时间 | 用户 | 状态 |
|-----|--------|------|---------|---------|------|------|
| 1234 | node | 2.1 | 156 | 3d 12h | sherlockguo | running |

### 2.2 排序与筛选

- 点击表头排序（CPU/内存/运行时间）
- 搜索框：模糊搜索进程名
- 最小化系统进程：可切换「仅显示用户进程」

### 2.3 核心进程高亮

自动识别并高亮关键进程（受 scripts 列表配置）：
- `node server.js`（WeMonitor）
- `node server.js`（WeMusic）
- `cloudflared`
- 其他可配置

### 2.4 自动刷新

5 秒刷新一次（进程数据变化快，刷新频率比系统指标高）。

## 3. 技术方案

### 3.1 数据采集

使用 `systeminformation.processes()` ：

```js
const si = require('systeminformation');
const data = await si.processes();
// data.list: Process[] { pid, name, cpu, mem, started, state, ... }
```

采集频率：每次页面请求时实时获取（不缓存），数据量小、命令执行快。

### 3.2 后端 API

`GET /api/v1/processes?sort=cpu&filter=node&userOnly=true`

- 参数可选
- 数据实时从 `systeminformation` 获取
- 返回前 50 条（避免数据量过大）

## 4. 安全考虑

- 进程列表展示的是所有用户的进程（通过 `systeminformation` 获取），是只读信息
- 不支持终止/重启进程操作（风险高，保持只读）
- 需要登录 + 激活才能访问

## 5. API 端点

| 方法 | 端点 | 说明 |
|------|------|------|
| `GET` | `/api/v1/processes` | 进程列表（支持 sort/filter/userOnly 参数） |
