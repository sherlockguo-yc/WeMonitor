# WeMonitor — 数据存储架构

> 状态：已实现

## 1. 概述

WeMonitor 使用 SQLite（better-sqlite3）作为唯一数据库，采用单文件存储、WAL 模式，支撑系统指标、服务配置、健康检查、用户认证等全部数据的持久化。

## 2. 双粒度存储

### 2.1 设计目标

原始数据采集频率为 30 秒～60 秒，如果永久保留原始数据，数据库体积会快速膨胀。通过时间聚合 + 定期清理实现高效存储。

### 2.2 存储层级

| 层级 | 表 | 粒度 | 保留时间 |
|------|-----|------|---------|
| 原始 | `metrics` | 1 分钟 | 7 天 |
| 聚合 | `metrics_hourly` | 1 小时 | 30 天 |

### 2.3 聚合逻辑

`lib/cleaner.js` 每 5 分钟执行一次增量聚合：

1. 扫描 `metrics` 表中已满的整点小时窗口
2. 计算 `avg_value` / `min_value` / `max_value` / `count`
3. 使用 `INSERT OR IGNORE` + UNIQUE 索引写入 `metrics_hourly`
4. 删除已聚合的原始数据
5. 删除超过 7 天的残留原始数据
6. 删除超过 30 天的聚合数据

### 2.4 全量补齐

首次启动时，`cleaner` 执行全量补齐（`runFull()`），将 `metrics` 表中所有历史数据聚合到 `metrics_hourly`，然后删除原始数据。

### 2.5 查询路由

指标查询 API 根据时间范围自动路由：

```
请求范围 > 1 小时 → metrics_hourly（性能优化）
请求范围 ≤ 1 小时 → metrics（精确数据）
```

## 3. 表结构

### 3.1 metrics（原始指标）

```sql
CREATE TABLE metrics (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  service TEXT NOT NULL,
  metric_name TEXT NOT NULL,
  labels TEXT DEFAULT '{}',
  value REAL NOT NULL,
  timestamp INTEGER NOT NULL
);
CREATE INDEX idx_metrics_lookup ON metrics(service, metric_name, timestamp);
```

### 3.2 metrics_hourly（聚合指标）

```sql
CREATE TABLE metrics_hourly (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  service TEXT NOT NULL,
  metric_name TEXT NOT NULL,
  labels TEXT DEFAULT '{}',
  avg_value REAL NOT NULL,
  min_value REAL NOT NULL,
  max_value REAL NOT NULL,
  count INTEGER NOT NULL,
  hour_ts INTEGER NOT NULL
);
CREATE UNIQUE INDEX idx_metrics_hourly_unique
  ON metrics_hourly(service, metric_name, labels, hour_ts);
```

### 3.3 services（被监控服务）

```sql
CREATE TABLE services (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL UNIQUE,
  scrape_url TEXT,
  scrape_interval INTEGER DEFAULT 30,
  health_type TEXT,
  health_target TEXT,
  enabled INTEGER DEFAULT 1
);
```

### 3.4 health_checks（健康检查）

```sql
CREATE TABLE health_checks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  service_id INTEGER REFERENCES services(id),
  status TEXT NOT NULL,
  latency_ms INTEGER,
  message TEXT,
  timestamp INTEGER NOT NULL
);
```

### 3.5 users（用户）

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'user',
  status TEXT NOT NULL DEFAULT 'pending'
);
```

### 3.6 backup_config（备份配置）

```sql
CREATE TABLE backup_config (
  service_name TEXT PRIMARY KEY REFERENCES services(name),
  enabled INTEGER DEFAULT 0,
  last_backup_at INTEGER,
  last_backup_size INTEGER,
  r2_path TEXT
);
```

## 4. 数据清理

`lib/cleaner.js` 负责：

1. **增量聚合**：每分钟检查一次，聚合完整的整点小时窗口
2. **过期清理**：删除超过保留期的原始数据和聚合数据
3. **事件裁剪**：部署事件日志保留最近 1000 条
4. **首次全量补齐**：启动时执行一次全历史聚合

## 5. 文件持久化

除 SQLite 外，部分数据通过 JSON 文件持久化：

| 文件 | 内容 |
|------|------|
| `data/wemonitor.db` | SQLite 主库 |
| `data/sessions.db` | Session 存储 |
| `data/cron-jobs.json` | 定时任务配置 |
| `data/cron-history.jsonl` | 定时任务运行历史 |
| `data/topology.json` | 网络拓扑配置 |
| `data/deploy-events.jsonl` | 部署事件流 |

## 6. 关键文件

| 文件 | 用途 |
|------|------|
| `lib/db.js` | SQLite 初始化 + 表结构创建 |
| `lib/cleaner.js` | 数据聚合与清理 |
| `lib/api/metrics.js` | 指标查询（含智能路由） |
| `config.js` | 保留期配置（`retentionDays` / `retentionHoursAgg`） |
