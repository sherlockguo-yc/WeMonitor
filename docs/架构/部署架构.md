# WeMonitor — 部署架构

> 状态：已实现

## 1. 概述

WeMonitor 通过 GitHub Actions CI/CD + N150 deploy-agent 实现全自动部署。流程为 `git push → CI 构建 → Release → webhook 通知 → deploy-agent 自动部署`。

## 2. CI/CD 流水线

```
git push master
  ↓
GitHub Actions (ubuntu-latest)
  npm ci → npm run build → 打包 wemonitor.tar.gz → GitHub Release (tag=sha)
  ↓  curl POST https://wemusic.sherlockguo.com/deploy
N150 webhook 接收
  ↓  写入部署队列
N150 deploy-agent (cron 每分钟)
  ↓  下载 → 校验 → rsync → restart
N150 ~/wemonitor/
  ├── server.js / lib/ / routes/ / views/ / public/
  ├── node_modules/（CI 预编译，不执行 npm ci）
  ├── data/（SQLite + sessions，rsync --exclude 保护）
  └── .version（部署版本号）
```

## 3. 部署代理

### 3.1 统一代理

所有项目（WeMusic / WeMonitor / WeDownload）共用 `~/deploy-agent.sh`，通过配置文件 `~/.deploy-projects.conf` 驱动：

```
wemonitor|sherlockguo-yc/WeMonitor|$HOME/wemonitor|18990|sha|wemonitor.tar.gz
wemusic|sherlockguo-yc/WeMusic|$HOME/wemusic|5174|sha|wemusic.tar.gz
wedownload|sherlockguo-yc/WeDownload|$HOME/wedownload|8080|sha|wedownload.tar.gz
```

格式：`PROJECT|REPO|DIR|PORT|TAG_MODE|FILE_NAME`

### 3.2 两种触发方式

| 方式 | 触发源 | 延迟 | 说明 |
|------|--------|------|------|
| Webhook | CI 构建完成后 `curl POST` | 秒级 | 60 秒内同项目只接受一次 |
| Cron | `*/1 * * * *` | ≤ 1 分钟 | 兜底：webhook 失败时最终也会部署 |

### 3.3 部署流程

1. 查询 GitHub API 获取最新版本（cron）或使用 webhook 传入版本
2. 与本地 `.version` 比对，相同则跳过
3. 下载 `.tar.gz`（`ghproxy.net` 镜像优先 → GitHub 直连兜底）
4. 版本校验失败 → 跳过镜像直连 GitHub 重试
5. 解压 → `rsync -a --delete --exclude 'data' --exclude '.env'`
6. 执行 `restart.sh`

### 3.4 健康检查自愈

cron 模式内置健康检查：`lsof -ti:$port` 检测端口，无响应时自动执行 `restart.sh`。

### 3.5 保护机制

- `data/` 和 `.env` 受 `rsync --exclude` 保护，不会被自动更新覆盖
- N150 上只有运行时文件，无 `src/` / `scripts/` 等源码目录

## 5. SSH 连接

| 方式 | 命令 | 说明 |
|------|------|------|
| Tunnel SSH | `ssh n150` | Cloudflare Tunnel SSH（`ssh.sherlockguo.com`），任何网络可连 |

## 6. 环境变量

N150 上 `~/wemonitor/.env`：
- `CF_API_TOKEN`：Cloudflare API Token（Tunnel 路由管理）
- `API_KEY`：Push 指标上报鉴权

由 `restart.sh` 在启动时 `source` 加载。

## 7. 服务重启

**禁止手动重启**。服务崩溃后 cron 健康检查在 1 分钟内自动恢复。如需立即更新部署，通过 `git push` 触发完整 CI → webhook 流程。

## 8. 关键文件

| 文件 | 位置 | 用途 |
|------|------|------|
| `.github/workflows/deploy.yml` | 仓库 | CI 构建配置 |
